commitbot

"I said I'd do that and forgot."

Capstone demo · MVP · May 2026

Problem statement.

What we set out to solve

Three interviews shaped this project — an APM, a UX designer, and a new support hire. They converged on the same finding.

"I'll send the report by Friday." Said, forgotten, missed.

The three observations

  • Workarounds exist, but none of them scale. Sticky notes work for one person until you have ten of them on your desk. Personal Slack channels capture commitments but no one else sees them. Notes apps don't ping the next person in a dependency chain. Slack stars and saved-for-later are invisible to teammates. Every interviewee had a patched-up system; every one of them broke under more load.
  • The hardest version of the problem isn't personal memory — it's cross-person visibility. When you tell a teammate "I'll send the report by Friday" and forget, the cost isn't just your own embarrassment; it's that the other person has no record either. They're stuck remembering what they're waiting on, from whom, and when to nudge.
  • People want capture to be ambient. Every interviewee said some version of "I don't want to open a new app to log this." The capture mechanism has to live inside the place where the commitment is made — overwhelmingly, Slack.
CommitBot's bet: make commitment-tracking ambient. Captured where you already are, surfaced to the right people, reminded without you having to babysit the reminder system.

System architecture.

One process, three layers

The whole system runs as a single Python process. One uvicorn invocation handles HTTP from the browser, webhooks from Slack, and the background scheduler — no separate web / bot / worker services to deploy.

The three-layer rule

Every piece of code lives in exactly one layer. A rule like "completing a commitment computes its outcome, bumps a version number, writes an edit-log row" exists in one function (commit_svc.mark_done). The Slack Done button and the dashboard Done button both call that function — no duplication, no drift.

Layer Files Job
Entry points slack_app.py, routes/* Parse the request, call a service, render a response. No business logic.
Services services/commitments.py, services/pings.py, services/reassignments.py Enforce every rule about what can happen to a commitment. No HTTP, no Slack.
Models models.py Pure SQLAlchemy declarations. No logic.

Tech stack.

Every choice with its rationale

The connecting theme: boring tech, picked deliberately. Every choice is the conventional one for its job.

Layer Choice Why
Language Python 3.13 Asked for in the brief. 3.14 has issues with pydantic-core's Rust bindings, so 3.13 is the floor.
Web framework FastAPI Async-ready. Handles Slack webhooks (which demand fast 3-second acks) and dashboard HTML routes in one app. Dependency injection eats auth + DB-session boilerplate.
Slack SDK slack-bolt Official SDK. Handles request-signature verification, the 3-second-ack rule, retries, lazy listeners.
ORM SQLAlchemy 2.0 Typed ORM. Editors autocomplete fields and catch type mistakes before runtime.
Database SQLite (one file) Zero setup. Trivial swap to Postgres later — change one env var. Single-file means easy backup and easy reset.
Scheduler APScheduler (in-process) No separate worker, no Redis, no message broker. Five recurring jobs run in the same Python process via a thread pool.
HTML templates Jinja2 + HTMX Server-rendered HTML with surgical row swaps. No SPA, no JS build step, no node_modules. Page loads are fast, the codebase is easy to debug.
Sessions itsdangerous Signs the session cookie. Same library Flask uses for its sessions. SameSite=Lax + Secure in production.
OAuth Sign in with Slack (OpenID Connect) Real auth, not a ?user_id= query param. State token verified with secrets.compare_digest, open-redirect guard on the next URL.
Migrations Hand-written, idempotent SQLite's ALTER TABLE is permissive enough. db.py:_apply_lightweight_migrations runs on every boot; safe to re-run.
Tests pytest 109 tests, ~3 seconds. Each fixture builds a fresh in-memory SQLite — full isolation per test.

Commitment lifecycle.

The state machine, plus outcomes

Every commitment lives in one of six states. Transitions are owned by the service layer and always write an audit-log row. 

                          ┌── put_on_hold ──► ON_HOLD ── resume ──┐
                          │                      ▲                │
                          │                      │ auto-resume    │
                          │                      │  (resume_at OR │
                          │                      │   deadline ≤ X)│

   created ──►   ACTIVE  ─┤                                       ▼
                          │                                  prior_state
                          │  request_reassignment                  │
                          ├──► ON_HOLD (reassignment limbo) ──┐    │
                          │                                   │    │
                          │       ┌── accept ──► REASSIGNED ◄─┤    │
                          │       ├── decline ─────────────────┤    │
                          │       ├── cancel  ─────────────────┤    │
                          │       └── expire 24h ──────────────┘    │
                          │                                         │
                          │  mark_done                              │

   Every terminal transition (COMPLETE / ARCHIVED / DELETED) stamps
   an OUTCOME:
      completed_at AND on time?  →  SUCCESS
      otherwise                  →  FAILED

States in plain language

StateWhat it meansPings?Editable?
ACTIVE You're working on it. YesYes
ON_HOLD Paused. Either you snoozed it, OR a reassignment is awaiting response. No Yes (unless awaiting reassignment)
REASSIGNED Handed to you by a teammate; you accepted. Functionally like ACTIVE but tagged so the chain is traceable. YesYes
COMPLETE You finished it. NoNo
ARCHIVED Completed and filed away for keeps. NoNo
DELETED In the 48-hour bin. Manual Purge button hard-deletes immediately; otherwise the scheduler does so at 48h. NoNo

Outcome — orthogonal to state

A commitment that ends up in any terminal state (COMPLETE / ARCHIVED / DELETED) is stamped with one of two outcomes:

  • SUCCESS — completed_at is set AND it's at-or-before the deadline (or there was no deadline).
  • FAILED — anything else: never completed, or completed late.

The rule lives in one place — compute_outcome in services/commitments.py — and every terminal transition calls it. Reopen or restore-to-active clears the outcome; restore-to-complete preserves it.

The prior_state trick: when a commitment moves into ON_HOLD (snooze or reassignment limbo), we stash where it came from. On resume / decline / cancel / expire, we restore to that state — so a REASSIGNED commitment that gets snoozed comes back as REASSIGNED, not ACTIVE.

User flows.

Three annotated journeys

A. Creating a commitment via /commit

You type:  /commit I'll send the report by Friday   in #general

  1. Slack POSTs to /slack/events
  2. slack-bolt routes to handle_commit_slash
  3. handler calls ack() within 3s (Slack's deadline)
  4. onboarding gate: _find_user + signed_in_at check
        └─ if not onboarded: ephemeral nudge, return
  5. handler posts the public channel message
        "@you committed: I'll send the report by Friday"
  6. handler calls commit_svc.create_commitment
        ├─ dedup check (workspace, channel, message_ts)
        ├─ default-priority resolution
        ├─ CommitmentRecipient rows for any @mentions
        └─ INSERT INTO commitments
        ↑ if anything fails, the channel post is deleted
          via chat.delete to avoid an orphan claim
  7. ping_svc.schedule_initial_ping
        └─ INSERT INTO pings (scheduled_for = now + base)
  8. threaded reply with a Done ✓ button
        (only the owner can press it — _is_commitment_owner)
  9. _refresh_home pushes updated Slack home view

Total round-trip: ~150 milliseconds.

B. The reassignment flow (Alice → Bob)

  ALICE                                          BOB
  ─────                                          ─────
  clicks Reassign on a commitment
    └─ modal opens: users_select + optional note

  commitment → ON_HOLD
    ├─ prior_state = ACTIVE (stashed)
    ├─ on_hold_resume_at = NULL
    ├─ pending pings deleted
    └─ Reassignment row: status = PENDING,
       expires_at = now + 24h

                                ──DM──►   "@alice wants to hand off
                                          'send the report' to you"
                                          [Accept] [Decline]

                                              Bob clicks Accept ──┘
                                              │
                                              ▼
                                ◄──DM──   Alice notified
                                              Reassignment.status = ACCEPTED
                                              Commitment.user_id   = Bob
                                              Commitment.state     = REASSIGNED
                                              Commitment.priority  = Bob's default
                                              fresh ping queued for Bob

  Alice's Reassigned tab          Bob's Active tab:
   shows the row, read-only,        shows the row, full action
   pill: "→ now with @Bob"           buttons, gets pings on his
                                     cadence

  ── alternatives ──

   Bob clicks Decline       → status=DECLINED, commitment → ACTIVE (Alice)
   Alice clicks Cancel      → status=CANCELLED, commitment → ACTIVE, Bob's DM retired
   24h with no response     → scheduler's expire_reassignments job:
                              status=EXPIRED, both parties DM'd

Key invariants enforced server-side: only owners can request / cancel; only the named target can accept / decline; targets must be onboarded; no edits allowed while pending; at most one pending reassignment per commitment.

C. The ping loop

TIME 0
  /commit just created the commitment. schedule_initial_ping ran.
  → Ping row created with scheduled_for = now + base_interval

EVERY 60 SECONDS
  Scheduler tick → process_due_pings:
    1. SELECT * FROM pings WHERE sent_at IS NULL AND scheduled_for <= now
    2. For each:
       a. If commitment state ∉ (ACTIVE, REASSIGNED): mark consumed, skip
       b. If user.global_pause:
            mark consumed, queue next ping (keep queue primed), skip
       c. deliver_ping → either log (dry run) or send_ping_dm
       d. db.flush() so the next-ping count query sees the just-sent one
       e. compute_next_ping_at(...) returns the time for the next ping
       f. INSERT INTO pings with scheduled_for = that time

INSIDE ESCALATION WINDOW  (now ≥ deadline − escalation_hours)
  Each ping's interval shrinks:
     interval = base / (rate^stages_so_far)   floored at max_ping_frequency

TIME D (deadline) and beyond
  Overdue mode — pings continue at floor cadence indefinitely
  unless the user has pressed Stop escalation (then base cadence forever).

Example with base=240m, rate=2, floor=30m: pings fire at 240, 120, 60, 30, 30, 30… as stages accumulate inside the escalation window.

D. Sign in with Slack

  1. User visits /  with no session
  2. required_user dependency raises LoginRequired
  3. Custom exception handler in main.py:
       - HTMX request? → 401 + HX-Redirect header
       - Otherwise   → 303 redirect to /auth/slack/login?next=…
  4. /auth/slack/login generates a 32-byte state token, stores it
     in the session, redirects to Slack's authorize endpoint
     (scope = openid profile email)
  5. User clicks Approve on Slack's page
  6. Slack → /auth/slack/callback?code=…&state=…
  7. State verified with secrets.compare_digest (timing-safe)
  8. POST openid.connect.token  → access_token
  9. GET  openid.connect.userInfo (Bearer access_token)
       → slack_user_id, slack_team_id, email, display_name
 10. Find-or-create the User row. Stamp signed_in_at = now.
 11. Sign session cookie (itsdangerous, 30 days, SameSite=Lax).
 12. views.publish fresh App Home so the user sees their
     commitments instantly when they switch back to Slack.
 13. 303 redirect to the saved next URL (defaults to /).

signed_in_at is the proof of onboarding — the Slack capture paths (/commit, message shortcut) refuse to log commitments for users without it.
