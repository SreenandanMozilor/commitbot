The Agent: how CommitBot auto-captures commitments

An end-to-end walkthrough of the opt-in AI pipeline that watches the messages you send in Slack, picks out personal commitments, and files them next to the ones you create by hand — with the same pings, dashboard, and audit trail.

1. TL;DR

When a user opts in, every message they send in a Slack channel where CommitBot is present gets quietly stashed in a buffer. Every 30 minutes the scheduler wakes up, hands the unprocessed buffer to a Claude classifier, and any verdict above the user's confidence floor (default 75%) becomes a real commitment — same row in the same table as anything created by /commit or the [[commit … ]] notation. Pings get scheduled, the dashboard picks it up, the Home tab shows it in a "Recently auto-captured" strip with a 60-minute Undo button. Anything the agent gets wrong is hard-deleted on Undo so it never pollutes the user's failed-commitments history.

Key invariants
  • The agent only looks at messages the user sends, never received messages.
  • It is off by default. Per-user opt-in via App Home.
  • It goes through the same create_commitment service-layer call as every other capture path — dedup, audit log, ping scheduling are identical.
  • False positives are reversed with a hard-delete, not a soft-delete, so outcome statistics stay honest.

3. A message's journey, step by step

Suppose Alice has opted in and types "I'll send the design doc by Friday EOD" in #design. Here's exactly what happens.

1. Slack delivers a message event
app/slack_app.py — Bolt event handler
Slack's Events API POSTs to /slack/events. Bolt validates the signing secret, deserialises the event, and dispatches to the @bolt_app.event("message") handler. The handler resolves the sender's User row and checks they're onboarded.
2. Message gets buffered (if opted in)
app/services/agent.py · buffer_message()
Before the existing notation-pattern match runs, the handler calls buffer_message(). It refuses to buffer if the user hasn't opted in, the text is empty, or the same (user, channel, ts) already exists (Slack retries can re-deliver the same event). The insert is wrapped in a SAVEPOINT so a duplicate-key collision can't roll back the surrounding handler — the notation capture, if any, still commits. 
with db.begin_nested():
    db.add(row)
    db.flush()    # IntegrityError → savepoint rolls back, return None
3. Scheduler ticks
app/scheduler.py · scan_for_commitments()
APScheduler fires every AGENT_SCAN_INTERVAL_MINUTES (default 30). It opens a fresh DB session and calls agent.scan_all(), which selects every user with agent_enabled = True and runs scan_user() for each one. Failures inside scan_user are caught per-user — one user's bad batch doesn't block anyone else's.
4. Buffer drain
app/services/agent.py · scan_user()
Up to 30 unprocessed rows are read in created_at order and shaped into HarvestedMessage objects. The batch size is bounded by the model's output-token budget — each verdict object is ~80 tokens, and the API call asks for 4096 max.
5. Classify
app/services/llm.py · AnthropicProvider.classify()
One Anthropic Messages API call per batch. The prompt is prefilled with [ so the model continues a JSON array without preamble; the response is glued back together and parsed. If the model truncates mid-array, a best-effort repair trims to the last complete } and closes the array — any partial verdict is dropped.
6. Apply the verdict
app/services/agent.py · loop body of scan_user()
Each row is processed inside its own SAVEPOINT so a race on one row can't roll back the others in the batch. If the verdict is below the user's confidence floor (default 75%) the row is marked processed and dropped. Otherwise:
  • Recipients are extracted from the message text via the same regex the notation handler uses (<@U…> Slack mentions plus plain @name).
  • Deadline hint from the LLM is parsed as ISO-8601 and rejected if it's in the past or more than a year out.
  • commit_svc.create_commitment(…, source=AGENT) creates the row. The same channel/ts dedup applies — if a notation got there first, the existing row is returned and not stomped.
  • agent_confidence and agent_rationale are stamped on the new row for the Home strip and dashboard.
  • schedule_initial_ping() queues the first Ping.
7. Home tab refresh
app/scheduler.py + app/slack_app.py · _refresh_home()
After the session commits, the scheduler calls views.publish for each user with at least one new capture. They see the row appear under Recently auto-captured with an Undo button.
8. Pings start firing
app/scheduler.py · process_due_pings()
The 60-second ping loop picks up the new Ping like any other, delivers it as a DM (or logs it under DRY_RUN_PINGS), then schedules the next one using the same cadence calculator that /commit captures use.

4. The classifier

The agent talks to an LLMProvider protocol — neither the buffer drain nor the service layer knows which implementation is on the other end. Two ship in-tree:

AnthropicProvider production

Uses the official anthropic SDK against claude-haiku-4-5 by default (configurable). One Messages API call per batch, prefilled with [ so the response is a clean JSON array. Budgeted at 4096 output tokens for a 30-message batch.

StubProvider dev/tests

Deterministic regex classifier — no network, no key. Matches patterns like "I'll …", "I will …", "I can pick up …" at calibrated confidences, with anti-patterns for hypotheticals, low-conviction qualifiers, and past tense. Used automatically when ANTHROPIC_API_KEY is empty.

The prompt, in one line

Three load-bearing pieces, kept terse because the prompt re-sends on every batch and verbosity is pure cost:

  1. Examples on both sides. Positive ("I'll send the spec by Friday") and negative ("I'll think about it", "Maybe I'll grab lunch", "John should fix that"). The negatives teach the false-positive shape — vague intent, hypothetical, third-party action, past tense.
  2. A JSON schema. Each verdict has message_id (echoed verbatim, used to match back to buffer rows — the service rejects any id we didn't send, so model hallucinations are filtered), is_commitment, confidence, rationale, deadline_hint, recipient_hints.
  3. Calibration bands. ≥0.90 = unambiguous; 0.75–0.90 = clear but fuzzy edges; 0.50–0.75 = real ambiguity, rationale should explain; <0.50 = not a commitment.

Prompt-injection containment

User messages are user-controlled input, so they go through json.dumps() before interpolation into the prompt. A message containing " or a hand-crafted JSON object can't close the prompt's quoted string and inject fake verdicts.

5. Safety rails

RailWhereWhat it does
Opt-in users.agent_enabled Defaults to FALSE. The buffer write is gated on it; toggle lives on App Home.
Dry-run AGENT_DRY_RUN Ships ON. Classifier runs, verdicts are logged, but no commitment is written and no buffer row is marked processed (so reruns are deterministic).
Confidence floor AGENT_CONFIDENCE_FLOOR + users.agent_confidence_floor_pct Default 0.75. Per-user override stored as a percent. Verdicts below the effective floor are dropped silently.
Hard-delete Undo undo_agent_capture() Within AGENT_UNDO_WINDOW_MINUTES (default 60), the inline Undo button hard-deletes the row. False positives don't end up in the user's failed-commitments stats.
Sender-only The system prompt + buffer_message() entry condition The handler buffers the sender's messages keyed on owner.agent_enabled. Messages received from other people are never classified.
Dedup with other capture paths UNIQUE(workspace, channel, ts) on commitments + find_existing_by_slack_message If a [[commit …]] notation already captured the same message, the agent path returns the existing row and skips stamping provenance.
Per-row savepoints scan_user() Each row's work is wrapped in db.begin_nested() so one race or DB hiccup can't roll back the batch's other successful captures.
Buffer retention AGENT_BUFFER_RETENTION_DAYS (default 7) Daily prune job removes raw message text after the window. Even with the agent off, we don't hoard Slack content.
Deadline guardrail _parse_deadline_hint() Rejects ISO-8601 hints that are in the past or more than 365 days out — better no deadline than a wrong one.

6. How users turn it on & use it

Turn it on

  1. Open CommitBot's App Home in Slack.
  2. Find the agent status strip at the top — it shows :zzz: Agent: off, the configured model, and the floor.
  3. Click Turn agent on. The chip flips to :robot_face: Agent: ON, a Scan recent messages button appears, and the buffer starts collecting.

Normal use

Type as you would in any Slack channel where CommitBot is invited. You don't need to use /commit, the [[…]] notation, or the right-click shortcut for the agent to work — but you can still use all of them. The agent is additive.

Review captures

Every 30 minutes (or sooner if you click Scan recent messages), any new auto-captures appear under Recently auto-captured on Home, sorted newest-first. Each row shows the captured text, the model's confidence percentage, the rationale, and three actions:

  • Undo (delete) — within 60 minutes. Hard-deletes the row.
  • Set deadline — opens the same date-time picker as for hand-captured commitments.
  • Mark done — completes it.
Important

The agent runs the buffer through the LLM in batches, not on every message — so a freshly-sent message won't show up as a capture instantly. Either wait for the next scheduled scan or hit Scan recent messages on Home.

7. Integration touchpoints

Five seams where the agent plugs into the rest of CommitBot. None of them are new code paths — the agent reuses what was already there.

SurfaceHow the agent uses it
Slack message events Existing Bolt handler that already powers notation capture. The agent inserts buffer_message() as a pre-step, gated on owner.agent_enabled.
Service layer (commit_svc.create_commitment) Single function called by every capture path — /commit, notation, shortcut, and now agent. Same dedup, same audit log (CommitmentEdit), same recipient handling.
Ping scheduler The agent calls schedule_initial_ping(), same helper the notation path calls. The 60s ping loop has zero knowledge of which capture path produced a Ping.
App Home New _home_agent_blocks() renders a status strip plus the recently-captured list. Action handlers agent_toggle, agent_scan_now, agent_undo::<id> handle clicks.
Dashboard Agent rows render with a distinct source pill AGENT alongside SLASH_COMMAND and NOTATION. Same filtering and tabs.
APScheduler Two new jobs: agent_scan (interval = AGENT_SCAN_INTERVAL_MINUTES) and agent_prune_buffer (daily).

Going live, the safe sequence

  1. Set ANTHROPIC_API_KEY in the deployment env.
  2. Leave AGENT_DRY_RUN=true. Opt one or two trusted users in and watch the logs — every classified row prints a one-line summary.
  3. Inspect a sample for false positives. Adjust the prompt if needed.
  4. Flip AGENT_DRY_RUN=false. Live captures begin on the next scan.

9. What gets stored

agent_message_buffer (new table)

Transient staging area for messages awaiting classification. Pruned after AGENT_BUFFER_RETENTION_DAYS. Unique on (user_id, slack_channel_id, slack_message_ts) so Slack event retries don't double-buffer. 

class AgentMessageBuffer:
  id                 # uuid pk
  user_id            # FK users.id, cascade
  slack_channel_id   "C…"
  slack_message_ts   "1700000000.000100"
  text               # truncated at 2000 chars on insert
  created_at
  processed_at       # non-null once scan has examined it

users additions

ColumnTypeMeaning
agent_enabledBOOLEAN NOT NULL DEFAULT 0Per-user opt-in.
agent_confidence_floor_pctINTEGER NULLOptional override of the system floor, stored as a percent (0–100).

commitments additions

ColumnTypeMeaning
sourceENUMNew value AGENT alongside SLASH_COMMAND, NOTATION, etc.
agent_confidenceFLOAT NULLVerdict confidence (0.0–1.0). Only set when source = AGENT.
agent_rationaleTEXT NULLOne-sentence rationale rendered as a context line on Home.
Migration

Schema evolves via _apply_lightweight_migrations() in app/db.py — no Alembic. Existing DBs get the new columns via additive ALTER TABLE on startup; fresh DBs have them created by Base.metadata.create_all(). Both paths are idempotent.

10. Operations & cost

Scheduled jobs the agent adds

  • agent_scan — every 30 minutes (configurable). Drains every opted-in user's buffer.
  • agent_prune_buffer — every 24 hours. Drops buffer rows older than the retention window.

What an API call costs (order-of-magnitude)

Each scan call for one user batches up to 30 messages into a single Anthropic request:

  • Input: ~120 tokens of system prompt + ~25 tokens × 30 messages ≈ ~900 tokens.
  • Output: ~80 tokens × 30 verdicts ≈ ~2400 tokens.

At Haiku-class pricing, that's a fraction of a cent per scan. A user who actually fills 30 messages between scans is unusual — most scans process 0–5 rows.

Failure modes & how the system responds

FailureWhat happens
Anthropic API down / rate-limitedAnthropicProvider catches the exception, returns []. No buffer rows are marked processed, so the next scan retries.
API key absent or invalidget_provider() logs and falls back to StubProvider. The agent keeps running on regex-quality verdicts.
Model returns truncated JSON_attempt_json_repair() trims to the last complete } and closes the array. Partial verdicts are dropped — their buffer rows remain unprocessed for the next sweep.
Model hallucinates a message_id we didn't sendFiltered against valid_ids, dropped silently.
Race between scheduler and "Scan now" buttonPer-row SAVEPOINT in scan_user isolates the colliding row; everything else commits cleanly.
Slack re-delivers a message eventUniqueConstraint on the buffer table catches it; buffer_message() returns None and the surrounding transaction is unaffected.

11. FAQ

Does the agent read messages that other people send to me?

No. The buffer entry is gated on the sender's agent_enabled flag, and the classifier prompt is explicit: "decide whether the SENDER is making a personal commitment." Even if a teammate types "I'll send it" in your DM, it's their flag that decides whether it gets buffered for them, not yours.

What channels does it watch?

Any channel CommitBot has been invited to and that sends message events to the bot. We don't pull history on demand — we react to live events. If you want the agent to see a channel, invite the bot.

Can the agent capture a message someone else marked with the notation, like [[commit …]]?

The same Slack message can match both the notation regex and the agent classifier. The first one to write wins by virtue of the UNIQUE(workspace, channel, ts) constraint. If the notation captures first, the agent's create_commitment call returns the existing row and the agent simply doesn't stamp its provenance on it — no duplicate, no overwrite.

What if the agent gets one wrong?

Within 60 minutes of capture, the row shows an inline Undo button on Home. Clicking it hard-deletes the commitment — it leaves no trace, doesn't count as a failed commitment, doesn't show in the Bin. After 60 minutes you can still soft-delete it from the dashboard, but it'll behave like any other deletion.

Why dry-run by default?

The first thing you want when you enable an LLM-driven feature on a real workspace is to see what it would do without it actually doing anything. Dry-run logs every verdict so you can inspect calibration and prompt quality before flipping the switch. Once you're satisfied, set AGENT_DRY_RUN=false.

What happens to my data if I turn the agent off?

The toggle just flips users.agent_enabled to false. New messages stop being buffered; the daily prune job removes any unscanned buffer rows after the retention window. Commitments already created stay — they're regular commitments at that point, indistinguishable from manual ones except for the source pill and the rationale text.

How do I add my own classifier?

Implement the LLMProvider Protocol (a single classify(messages) method returning a list of ClassifiedCandidate), register it in get_provider(), and point AGENT_PROVIDER at it. Neither the agent service nor the schedulers need to change.

Where do I look in the code first?

Start with app/services/agent.py — the orchestration is ~400 lines and reads top-to-bottom. From there: app/services/llm.py for the prompt and providers, app/slack_app.py for the message handler and Home UI, app/scheduler.py for the periodic jobs. tests/test_agent.py exercises the contract end-to-end against the stub provider.
